# /learn-context — 챗봇 정보 수집 및 컨텍스트 구성

## 목적
PM이 제공하는 챗봇 관련 자료(파일, URL)를 분석하고, 웹 리서치를 병행하여 context/ 파일을 구성한다.
PM과의 대화를 최소화하고, 가용한 정보로부터 최대한 추론하여 가안을 먼저 제시한다.

---

## Step 0. 사전 확인

- `memory/MEMORY.md`를 읽고 이전 세션 상태를 파악한다.
- `context/`에 이미 작성된 파일이 있는지 확인한다. 있다면 신규 작성이 아닌 갱신 모드로 진행한다.
- `inputs/index.md`를 읽고 기존에 등록된 자료를 파악한다.

---

## Step 1. 입력 수집

PM에게 다음 정보를 요청한다:

- **서비스명**: 챗봇 또는 서비스의 이름
- **간단한 소개**: 어떤 챗봇인지 1~2문장 설명 (예: "이커머스 고객 상담용 AI 챗봇", "영어 회화 연습 튜터봇")
- **관련 자료** (파일 또는 URL 중 최소 하나 필수):
  - **파일**: `inputs/` 디렉토리에 업로드 요청 (PRD, 기획서, API 스펙, 고객 여정 맵 등)
  - **URL**: 챗봇 서비스 사이트, 랜딩 페이지, 도움말 페이지 등

PM이 자료 입력을 완료했다고 확인하면 다음 단계로 진행한다.
새 파일이 추가되었으면 `inputs/index.md`를 업데이트한다.

---

## Step 2. 입력 분석 + 웹 리서치

### a) 입력 분석
`inputs/`의 모든 파일과 제공된 URL을 읽고, 아래 항목별로 추출 가능한 정보를 식별한다.

#### 필수 항목

| 항목 | 설명 | 추출 힌트 |
|------|------|-----------|
| **Chatbot Identity** (챗봇 정체성) | 챗봇 이름/서비스명, 유형(CS, FAQ, AI 튜터 등), 서비스 도메인(이커머스, 의료, 교육 등). 모든 분석의 출발점. | PRD 서두, 랜딩 페이지 헤드라인, 서비스 소개 섹션 |
| **Target Users** (주요 사용자) | 챗봇을 사용하는 주요 사용자층과 그들의 목적/니즈. 현실적 시나리오 생성의 기반. | PRD의 타겟 고객 섹션, 고객 여정 맵의 페르소나, 랜딩 페이지의 "누구를 위한" 문구 |
| **Core Capabilities** (핵심 기능) | 챗봇이 처리할 수 있는 주요 기능 목록. 테스트 대상 정의. | PRD 기능 명세, API 스펙의 엔드포인트 목록, 도움말 페이지의 "할 수 있는 것" 섹션 |
| **Out-of-Scope** (범위 밖 영역) | 챗봇이 처리하지 않는/못하는 영역, 의도적으로 제외한 기능. 엣지 케이스·경계 테스트의 핵심. | PRD의 제약 사항/비기능 요구사항, FAQ의 "지원하지 않는" 항목. **inputs/에 명시되지 않을 확률이 높음 → 웹 리서치 기반 추측 필요** |
| **User Flows** (유저 플로우) | 사용자가 챗봇과 대화하는 주요 흐름 (예: 상황 설명 → 상품 추천 → 가격 문의 → 구매 안내). 다수의 플로우가 존재할 수 있음. | 고객 여정 맵, PRD의 사용 시나리오, User Logs(있을 경우). **직접 추출이 어려운 경우 → 도메인 + Core Capabilities 기반 추측 필요** |

#### 선택 항목

| 항목 | 설명 | 추출 힌트 |
|------|------|-----------|
| **API Spec** (API 스펙) | 챗봇 호출을 위한 API 명세. Endpoint URL, 요청/응답 포맷, 인증 방식, 세션 관리(멀티턴 대화 ID), 응답 방식(일반/스트리밍), 파일 업로드 지원 여부 등을 포함. 시뮬레이션 실행의 필수 전제 조건. | API 문서, Swagger/OpenAPI 스펙, 기술 가이드, Postman 컬렉션 |
| **User Logs** (운영 데이터) | 실제 사용자-챗봇 대화 로그. 유저 플로우 정밀화, 실제 패턴 발견에 활용. | 대화 로그 CSV/JSON, 운영 리포트, CS 상담 기록 |
| **Issues** (이슈) | 파악된 문제점 + 웹 검색으로 발견된 고객 불만/취약점. 고가치 TC 직접 타겟팅 가능. | QA 리포트, 버그 트래커, 고객 불만 내역. **웹 검색으로 수집한 리뷰/불만도 여기에 포함 (PM 확인 필수)** |

### b) 웹 리서치

Step 2a의 입력 분석에서 정보가 부족한 항목을 웹 검색으로 보완한다.
리서치는 3단계로 진행하며, 앞 단계의 결과가 뒷 단계의 검색 키워드에 반영된다.

> **서비스 규모에 따른 전략 분기**
> - **규모가 큰 서비스**: 서비스명으로 직접 검색 (예: "카카오톡 채널 챗봇")
> - **규모가 작은 서비스**: 동일 도메인의 유명 서비스를 기준으로 검색하여 유사 패턴 분석 (예: 교육 도메인 → "뤼이드 AI 튜터", "산타토익 챗봇")

#### Phase 1. 정체성 + 포지셔닝 (Chatbot Identity)

**목적**: 서비스의 정체성, 시장 포지셔닝, 공식 소개를 파악한다. 이후 Phase의 모든 검색 키워드가 여기서 확정된 서비스명/도메인에 기반하므로 반드시 먼저 수행한다.

| 검색 대상 | 검색 키워드 예시 | 기대 수확 |
|-----------|-----------------|-----------|
| 공식 소개·보도자료 | `"{서비스명}" 출시 OR 런칭 OR 소개` | 서비스 유형, 도메인, 공식 포지셔닝 |
| 시장 분석·비교 기사 | `"{도메인}" 챗봇 비교 OR 순위 OR 시장` | 경쟁 서비스 목록, 차별점, 시장 내 위치 |
| 기업/서비스 소개 페이지 | 직접 URL 방문 (랜딩 페이지, About 페이지) | 공식 기능 소개, 슬로건, 타겟 고객 문구 |

**저장**: `knowledge/identity-{서비스명-slug}.md`

#### Phase 2. 기능 + 사용자 (Target Users, Core Capabilities)

**목적**: 서비스가 누구를 대상으로 무엇을 할 수 있는지 파악한다.

| 항목 | 검색 대상 | 검색 키워드 예시 | 기대 수확 |
|------|-----------|-----------------|-----------|
| **Target Users** | 사용자 리뷰·사용 후기 | `"{서비스명}" 사용 후기 OR 리뷰 OR 사용해보니` | 실제 사용자 유형, 사용 목적, 만족/불만 포인트 |
| | 시장·타겟 분석 | `"{도메인}" 사용자 OR 고객층 OR 타겟` | 도메인 일반적 사용자 특성, 연령대, 니즈 |
| **Core Capabilities** | 기능 소개·도움말 | `"{서비스명}" 기능 OR 사용법 OR 도움말 OR 가이드` | 공식 기능 목록, 사용 가이드 |
| | 기능 비교 | `"{서비스명}" vs OR 비교 OR 장단점` | 경쟁사 대비 기능 차별점 |

> API Spec은 고객사가 제공하는 실제 QA 서버 API 명세이므로 웹 검색 대상이 아니다.

**저장**:
- `knowledge/users-{서비스명-slug}.md`
- `knowledge/capabilities-{서비스명-slug}.md`

#### Phase 3. 한계 + 흐름 + 이슈 (Out-of-Scope, User Flows, Issues)

**목적**: Phase 1-2에서 파악한 "서비스가 하는 것"을 기반으로 "하지 않는 것", "사용자가 실제로 경험하는 흐름", "문제점"을 파악한다.

| 항목 | 검색 대상 | 검색 키워드 예시 | 기대 수확 |
|------|-----------|-----------------|-----------|
| **Out-of-Scope** | 부정 리뷰·불만 | `"{서비스명}" 안되는 OR 불가능 OR 지원안함 OR 아쉬운` | 사용자가 기대했으나 없는 기능 |
| | 경쟁사 기능 비교 | `"{경쟁서비스명}" 기능` (Phase 1에서 확인한 경쟁사) | 경쟁사에는 있으나 대상 서비스에 없는 기능 |
| | 도메인 표준 기능 | `"{도메인}" 챗봇 필수 기능 OR 기본 기능` | 도메인에서 보편적으로 기대되는 기능 |
| **User Flows** | 사용 가이드·튜토리얼 | `"{서비스명}" 사용법 OR 튜토리얼 OR 시작하기 OR 가이드` | 공식 권장 사용 흐름 |
| | 사용자 경험 후기 | `"{서비스명}" 사용 경험 OR 후기 OR 대화 OR 상담` | 실제 대화 패턴, 사용자 여정 |
| | 도메인 일반 흐름 | `"{도메인}" 챗봇 대화 흐름 OR 시나리오 OR UX` | 도메인 표준 대화 패턴 |
| **Issues** | 고객 불만·리뷰 | `"{서비스명}" 불만 OR 오류 OR 버그 OR 문제` | 알려진 버그, 반복 불만 |
| | 커뮤니티·포럼 | `"{서비스명}" site:reddit.com OR 커뮤니티 OR 카페` | 사용자 커뮤니티 발 이슈 |

**저장**:
- `knowledge/out-of-scope-{서비스명-slug}.md`
- `knowledge/user-flows-{서비스명-slug}.md`
- `knowledge/issues-{서비스명-slug}.md`

#### 웹 리서치 공통 규칙

- 모든 knowledge/ 파일에는 **검색일, 출처 URL, 검색 키워드**를 명시한다.
- Phase별로 검색 결과가 부족하면 키워드를 변형하여 재검색한다 (동의어, 영문 키워드 등).
- 키워드 변형 후에도 결과가 부족한 항목은 에이전트가 도메인 지식과 유사 서비스 패턴을 기반으로 자체 판단하여 [추측]으로 가안에 포함한다 (knowledge/ 파일에 판단 근거를 기록).
- `knowledge/index.md`에 생성된 모든 파일을 등록한다.
- **slug 규칙**: 서비스명을 영문 소문자+하이픈으로 변환한다 (예: "카카오 채널" → `kakao-channel`).

---

## Step 3. 가안 작성

필수 5개 항목 + 해당되는 선택 항목에 대해 가안을 작성한다.

각 정보에 출처 태그를 부여한다:
- **[추출]**: `inputs/`에서 직접 추출. 해당 파일 경로 표시
- **[추측]**: `knowledge/`의 웹 검색 결과 기반 추론. 해당 파일 경로 표시

**핵심 원칙: "묻지 말고, 먼저 제안하라"**
정보가 부족해도 직접 질문하지 않는다. 가용한 정보와 웹 리서치 결과를 최대한 활용하여 가안을 작성하고, PM에게 확인을 받는다.

---

## Step 4. PM 확인

가안 전체를 한 번에 PM에게 제시한다 (Plan 형태로 CLI 표시).

- **[추출] 항목**: PM이 검토 (수정 가능)
- **[추측] 항목**: PM 확인 필수 — 승인, 수정, 또는 삭제

PM의 피드백을 반영하여 가안을 수정한다.
이 과정에서 불명확한 부분이 있으면 이때 질문한다.

---

## Step 5. context/ 파일 생성

PM 확인이 완료되면 context/ 파일을 생성한다.

- 항목별로 별도 파일 또는 통합 파일로 작성 (내용 규모에 따라 판단)
- 각 파일 하단에 `## Sources` 섹션 필수
- `context/index.md`에 @import 추가하여 세션 시작 시 자동 로딩되도록 한다

---

## Step 6. 메모리 업데이트

- `memory/task-log.md`에 작업 기록
- `memory/lessons-learned.md`에 교훈 기록 (있을 경우)
- `memory/MEMORY.md`의 프로젝트 현황 갱신
