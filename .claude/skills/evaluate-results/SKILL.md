# /evaluate-results — 시뮬레이션 결과 평가

## 목적
시뮬레이션 실행 결과(`outputs/simulations/{run-id}/`)를 TC에 정의된 평가 기준과 PM이 추가한 공통 평가 기준에 따라 평가한다.
에이전트 자신이 LLM-as-judge로서 챗봇 응답을 판정하며, 모든 판정에는 근거를 명시한다.

---

## Step 0. 사전 확인

- `memory/MEMORY.md`를 읽고 이전 세션 상태를 파악한다.
- `context/` 파일들을 읽고 대상 챗봇을 숙지한다.
- `outputs/simulations/`에 시뮬레이션 결과가 있는지 확인한다. 없으면 `/run-simulation`을 먼저 실행하도록 안내한다.
- 시뮬레이션 디렉토리가 여러 개인 경우, run-id 기준 최신 순으로 정렬하여 목록을 준비한다.
- 최신 run의 `summary.md`의 `## 평가 상태` 또는 `## 평가 결과` 섹션을 확인한다. 이미 평가 완료된 run이면 PM에게 "재평가 여부"를 확인한다.

---

## Step 1. 평가 설정

PM에게 아래 3가지를 **한 번의 메시지**로 묶어 제시한다. 각 항목에 기본값을 명시하여 PM이 "기본값으로 진행"으로 빠르게 승인할 수 있도록 한다.

### 1a) 평가 대상 선택

| 항목 | 기본값 | 옵션 |
|------|--------|------|
| 시뮬레이션 run | 최신 run-id | PM이 다른 run-id 지정 가능 |
| TC 범위 | 전체 (해당 run의 모든 tc-*.md) | 특정 TC 지정 (예: TC-001, TC-005) |

- 기본 제시 형식: `"{run-id}"의 전체 TC {N}개를 평가합니다. (completed: {N}, partial: {N}, error: {N})`
- `error` 상태 TC도 평가 대상에 포함한다 (Turn 1부터 실패한 경우 "평가 불가"로 처리 — Step 2 에러 TC 처리 규칙 참조).

### 1b) 공통 평가 기준

PM에게 TC에 정의된 평가 기준 외에 **모든 TC에 공통으로 적용할 추가 기준**이 있는지 확인한다.

제시 형식:
> TC별 평가 기준은 이미 정의되어 있습니다. 이 외에 모든 TC에 공통으로 적용할 추가 평가 기준이 있으신가요?
> 예시: 톤앤매너 일관성, 응답 길이 적절성, 안전성/할루시네이션 체크, 한국어 자연스러움 등
> 없으면 "없음"으로 진행합니다.

- PM이 "없음"이면 TC 정의 기준만으로 평가한다.
- PM이 추가 기준을 제시하면, 이를 `공통 평가 기준`으로 등록한다. 각 기준은 TC 평가 기준과 동일하게 **예/아니오로 판단 가능한 체크리스트 형태**로 정리한다. PM이 모호하게 제시하면 (예: "톤이 괜찮은지 봐주세요") 에이전트가 구체적 기준으로 변환하여 PM 확인을 받는다.
  - 예: "톤이 괜찮은지" → `- [ ] 챗봇 응답이 존댓말(~입니다/~합니다)을 일관되게 사용하는가`
- 공통 평가 기준은 **모든 TC에 동일하게 적용**된다 (TC별 차등 적용 불가).

### 1c) 엄격도 및 특별 초점

| 항목 | 기본값 | 옵션 |
|------|--------|------|
| 엄격도 | 보통 | 관대 / 보통 / 엄격 |
| 특별 초점 | 없음 | PM이 지정 (예: "할루시네이션에 집중", "응답 누락 위주로") |

**엄격도 정의**:

| 엄격도 | 판정 기준 | 적용 상황 |
|--------|-----------|-----------|
| 관대 | 핵심 의미가 전달되면 Pass. 사소한 누락·표현 차이는 무시 | 초기 프로토타입, 탐색적 테스트 |
| 보통 | TC 기대 응답의 핵심 내용이 모두 포함되어야 Pass. 표현 차이는 허용하되 내용 누락은 Partial | 일반 QA |
| 엄격 | 핵심 내용 + 부가 품질(정확성, 완전성, 톤)까지 충족해야 Pass. 사소한 누락도 Partial | 출시 전 최종 검증, 규제 대상 도메인 |

- PM이 별도 지정하지 않으면 "보통 / 특별 초점 없음"으로 진행한다.

### 설정 확인 후

PM 답변을 받으면 아래를 명시적으로 고정한다:

```
- 평가 대상: {run-id}, TC {범위} ({N}개)
- 공통 평가 기준: {목록 또는 "없음"}
- 엄격도: {관대/보통/엄격}
- 특별 초점: {내용 또는 "없음"}
```

PM 확인 후 평가를 시작한다.

## Step 2. 평가 실행 (LLM-as-judge)

에이전트가 각 TC의 시뮬레이션 결과를 읽고, 평가 기준에 따라 판정한다. 외부 API 호출 없이 에이전트 자신이 판정 주체다.

### 평가 단위

TC 단위로 평가한다. 각 TC에 대해 아래 세 가지를 순차로 수행한다:

1. **턴별 응답 분석** (보조 지표)
2. **TC 평가 기준 판정** (주 지표)
3. **공통 평가 기준 판정** (PM 지정 시)

### 2a) 턴별 응답 분석 (보조 지표)

각 턴에 대해 TC의 기대 응답과 실제 챗봇 응답을 비교 분석한다. 이 분석은 TC 평가 기준 판정의 **근거 자료**이며, 그 자체로 Pass/Fail을 결정하지 않는다.

턴별 분석 항목:

| 항목 | 내용 |
|------|------|
| 기대 응답 충족도 | 기대 응답의 핵심 내용 각각에 대해 실제 응답에 포함 여부를 판단 |
| 추가 관찰 | 기대 응답에 없지만 주목할 만한 응답 특성 (예: 할루시네이션, 불필요한 정보, 톤 이탈) |
| TC flow 이탈 영향 | 해당 턴에서 사용자 메시지가 조정(re-anchor)된 경우, 조정이 평가에 미치는 영향을 기술 |

**TC flow 이탈 처리 규칙**:
- 실전송 메시지가 TC 원본과 다른 턴은, 챗봇이 기대 흐름에서 벗어났다는 신호다.
- 이탈 자체를 자동 감점하지 않는다. 다만 이탈로 인해 TC의 원래 테스트 목적이 검증 불가능해진 경우, 해당 평가 기준을 "판정 불가 (flow 이탈)"로 표기한다.
- 이탈 후에도 챗봇이 적절히 대응했다면 (예: 사용자 재질문에 올바르게 답변) 해당 턴의 응답 자체는 긍정적으로 평가할 수 있다.

### 2b) TC 평가 기준 판정 (주 지표)

TC에 정의된 `### 평가 기준` 체크리스트의 각 항목을 판정한다.

**기준별 판정**:

| 판정 | 의미 | 표기 |
|------|------|------|
| Pass | 해당 기준을 충족 | `- [x]` |
| Fail | 해당 기준을 미충족 | `- [ ]` |
| 판정 불가 | flow 이탈 등으로 해당 기준을 검증할 수 없음 | `- [~]` |

- 각 판정에는 **1줄 이상의 근거**를 반드시 기술한다. 근거는 실제 챗봇 응답의 구체적 내용을 인용하여 작성한다.
  - Pass 예: `- [x] 상품 추천 시 3개 이상의 옵션을 제시하는가 — Turn 2에서 4개 상품(A, B, C, D)을 추천함`
  - Fail 예: `- [ ] 가격 정보를 포함하여 답변하는가 — Turn 3에서 상품명만 언급하고 가격 정보 누락`
  - 판정 불가 예: `- [~] 후속 질문에 일관된 맥락을 유지하는가 — Turn 2에서 flow 이탈로 re-anchor 발생, 원래 맥락 검증 불가`

### 2c) 공통 평가 기준 판정

Step 1b에서 PM이 추가한 공통 평가 기준이 있는 경우, TC 평가 기준과 동일한 방식으로 판정한다.

- 공통 평가 기준은 TC 평가 기준과 **별도 섹션**으로 기록한다 (포맷은 Step 3 참조).
- 판정 방식 (Pass/Fail/판정 불가)과 근거 기술 규칙은 2b와 동일하다.

### TC 레벨 최종 판정

TC 평가 기준 + 공통 평가 기준의 **전체 판정 결과를 종합**하여 TC 레벨 판정을 내린다.

| TC 판정 | 조건 |
|---------|------|
| **Pass** | 모든 기준이 Pass (판정 불가 항목은 제외하고 나머지 전체가 Pass) |
| **Fail** | 모든 기준이 Fail (판정 불가 제외) |
| **Partial** | Pass와 Fail이 혼재 |
| **판정 불가** | 모든 기준이 판정 불가 (전체 flow 이탈 등 극단적 경우) |

- 판정 불가 항목이 포함된 경우: 판정 불가 항목을 **제외**한 나머지 기준으로 TC 레벨을 결정한다. 단, 판정 불가 비율이 50% 이상이면 TC 레벨에 "(제한적 평가)" 주석을 추가한다.
- `error` 상태 TC (Turn 1부터 실패): TC 레벨 = `평가 불가 (실행 실패)`. 평가 기준 판정을 수행하지 않는다.
- `partial` 상태 TC (일부 턴만 완료): 완료된 턴까지의 응답으로 판정 가능한 기준만 평가한다. 나머지는 판정 불가로 처리.

### 진행 표시

TC 평가 중 진행 상황을 PM에게 표시한다:
- `[3/15] TC-003 평가 중...`
- 평가 완료 시 즉시 판정 결과 표시: `[3/15] TC-003 → Pass`

## Step 3. 결과 저장

### 3a) TC별 결과 파일 갱신 (`tc-{번호}.md`)

기존 `## 평가 대기` 섹션 전체를 `## 평가 결과` 섹션으로 **교체**한다. `## 평가 대기` 위의 내용(참조, 실행 결과, 대화 기록)은 변경하지 않는다.

**교체 후 포맷**:

```markdown
## 평가 결과

- **TC 판정**: {Pass / Partial / Fail / 판정 불가 / 평가 불가 (실행 실패)}
- **엄격도**: {관대 / 보통 / 엄격}
- **평가 시각**: {ISO 8601}

### 턴별 응답 분석

#### Turn 1
- **기대 응답 충족도**:
  - {핵심 내용 1}: 충족 — {근거}
  - {핵심 내용 2}: 미충족 — {근거}
- **추가 관찰**: {관찰 사항 또는 "없음"}
- **TC flow 이탈 영향**: {영향 기술 또는 "해당 없음"}

#### Turn 2
(동일 포맷)

### TC 평가 기준
- [x] {기준 1} — {근거}
- [ ] {기준 2} — {근거}
- [~] {기준 3} — {근거}

### 공통 평가 기준
- [x] {공통 기준 1} — {근거}
- [ ] {공통 기준 2} — {근거}

(PM이 공통 평가 기준을 지정하지 않은 경우 이 섹션 생략)

### TC 기대 응답 (참조용)
{기존 내용 유지 — TC에서 복사한 턴별 기대 응답}
```

**재평가 시**: 이전 `## 평가 결과` 섹션을 새로운 결과로 덮어쓴다. 이전 평가 이력은 보존하지 않는다 (필요 시 git 이력에서 복원 가능).

### 3b) 실행 요약 파일 갱신 (`summary.md`)

기존 `## 평가 상태` 섹션(`미완료`)을 아래로 **교체**한다.

**교체 후 포맷**:

```markdown
## 평가 결과

### 평가 정보
- **평가 시각**: {ISO 8601}
- **평가 범위**: {전체 / TC-001, TC-005 등}
- **엄격도**: {관대 / 보통 / 엄격}
- **특별 초점**: {내용 또는 "없음"}
- **공통 평가 기준**: {목록 또는 "없음"}

### 판정 통계

| 판정 | TC 수 | 비율 |
|------|-------|------|
| Pass | {N} | {%} |
| Partial | {N} | {%} |
| Fail | {N} | {%} |
| 판정 불가 | {N} | {%} |
| 평가 불가 (실행 실패) | {N} | {%} |
| **합계** | **{N}** | **100%** |

### TC별 판정 결과

| TC | 제목 | 판정 | Pass 기준 | Fail 기준 | 판정불가 기준 | 주요 소견 |
|----|------|------|-----------|-----------|-------------|-----------|
| TC-001 | {제목} | Pass | 3/3 | 0/3 | 0/3 | — |
| TC-002 | {제목} | Partial | 2/4 | 1/4 | 1/4 | Turn 2 가격 정보 누락 |
| TC-003 | {제목} | 평가 불가 | — | — | — | Turn 1 실행 실패 (HTTP 500) |

### 종합 큐레이션

#### 전체 품질 요약
{챗봇의 전반적 품질 수준을 1~2문장으로 요약. 판정 통계를 바탕으로 하되, 숫자 나열이 아닌 해석을 제공한다.}

#### 강점
- {구체적 강점 1 — 어떤 TC/턴에서 확인되었는지 명시}
- {구체적 강점 2}

#### 약점 및 개선 필요 사항
- {구체적 약점 1 — 어떤 TC/턴에서 확인되었는지, 어떤 평가 기준이 Fail인지 명시}
- {구체적 약점 2}

#### 공통 패턴
- {여러 TC에서 반복 관찰된 패턴 — 예: "멀티턴 대화에서 3턴 이후 맥락 유지 실패 경향"}
- {TC flow 이탈이 잦은 경우 그 패턴도 여기에 기술}

#### PM 권고 사항
- {평가 결과에 기반한 후속 조치 제안 — 예: "가격 정보 응답 로직 점검 권장", "TC-005~007 재설계 후 재시뮬레이션 권장"}
```

**재평가 시**: 이전 `## 평가 결과` 섹션을 새로운 결과로 덮어쓴다.

---

## Step 4. PM 보고

평가 완료 후 PM에게 핵심 결과를 보고한다:

- **판정 통계**: Pass/Partial/Fail/판정불가 분포 (한 줄 요약)
- **Fail 및 Partial TC 목록**: TC 번호, 제목, 주요 Fail 기준 (테이블 형태)
- **종합 큐레이션 요약**: 강점 1~2개, 약점 1~2개, 핵심 권고 사항
- **결과 파일 위치**: `outputs/simulations/{run-id}/summary.md`

PM이 결과를 보고 추가 질문이나 재평가 요청을 할 수 있다:
- **기준 조정 후 재평가**: Step 1로 돌아가 엄격도·공통 기준을 변경하여 재평가 가능
- **특정 TC 재평가**: 특정 TC만 지정하여 재평가 가능
- **결과 수용**: PM이 수용하면 Step 5(메모리 업데이트)로 진행

## Step 5. 메모리 업데이트

- `memory/task-log.md`에 작업 기록 (평가 대상 run-id, TC 수, 판정 분포, 사용한 엄격도)
