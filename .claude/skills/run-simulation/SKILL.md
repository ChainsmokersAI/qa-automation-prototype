# /run-simulation — 테스트 시뮬레이션 실행

## 목적
PM이 승인한 TC를 실제 챗봇 API에 실행하여, 챗봇의 실제 응답을 기록한다.
이 스킬은 **실행과 기록**만 담당한다. 기대 응답 대비 평가(합격/불합격 판정)는 별도 단계에서 수행한다.

---

## Step 0. 사전 확인

- `memory/MEMORY.md`를 읽고 이전 세션 상태를 파악한다.
- `context/` 파일들을 읽고 대상 챗봇을 숙지한다.
- `outputs/testcases/`에 승인된 TC가 있는지 확인한다. 없으면 `/generate-testcases`를 먼저 실행하도록 안내한다.
- `context/`에서 API Spec 정보를 확인한다 (Step 1에서 충분성 검증).

---

## Step 1. API Spec 충분성 검증

아래 기준에 따라 context/의 API Spec 정보를 검증한다.

| 등급 | 항목 | 설명 | 미충족 시 대응 |
|------|------|------|---------------|
| **필수** | Endpoint URL | 챗봇 API의 호출 주소 | 시뮬레이션 불가 — PM에게 보충 요청 |
| **필수** | 요청 포맷 | HTTP method, body 구조, 메시지 필드명 | 시뮬레이션 불가 — PM에게 보충 요청 |
| **필수** | 응답 포맷 | 응답 body에서 챗봇 메시지를 추출하는 방법 | 시뮬레이션 불가 — PM에게 보충 요청 |
| **권장** | 인증 방식 | API Key, Bearer Token 등 | 인증 불요 API로 간주. 인증 에러 발생 시 PM에게 요청 |
| **권장** | 세션 관리 | 멀티턴 대화를 위한 conversation_id 관리 방법 | 멀티턴 TC는 실행 불가로 표시(skipped). 싱글턴 TC만 실행 |
| **권장** | 응답 방식 | 일반(동기) 또는 스트리밍(SSE 등). 스트리밍 시 청크 구분 방식 | 일반 응답으로 시도. 응답 헤더에서 스트리밍 감지되면(Content-Type: text/event-stream 등) 자동 전환 |
| **선택** | 파일 업로드 | 이미지/문서 첨부 방법 (multipart 등) | 파일 첨부 관련 TC는 스킵. 텍스트 전용 TC만 실행 |
| **선택** | Rate Limit | 호출 빈도 제한 | 보수적 간격(요청 간 1초 대기)으로 실행 |
| **선택** | 에러 코드 | API 에러 응답 형식과 의미 | HTTP 상태코드로 성공/실패 판단 |

**판정**: 필수 3항목이 모두 있으면 "시뮬레이션 가능", Step 2로 진행. 하나라도 없으면 PM에게 부족 항목을 명시하여 보충을 요청한다 (API 문서, Swagger 스펙, cURL 예시 등 제공 가능). 보충이 완료되면 context/ API Spec 정보를 갱신한 뒤 진행한다.

## Step 2. TC 선택

PM에게 실행할 TC 범위를 확인한다:

- **전체 실행**: 모든 승인된 TC 실행
- **시나리오별 실행**: 특정 시나리오의 TC만 실행 (예: S-001의 모든 TC)
- **개별 실행**: 특정 TC만 지정 (예: TC-001, TC-005)

PM이 별도 지정하지 않으면 전체 실행한다.

### 실행 전 요약

선택된 TC 목록을 PM에게 제시한다:
- TC 수, 대화 유형별 분포 (싱글턴 N개, 멀티턴 N개)
- 세션 관리 미확인으로 스킵되는 멀티턴 TC가 있으면 명시
- 예상 API 호출 수 (전체 턴 수 합산)

PM 확인 후 실행을 시작한다.

---

## Step 3. 시뮬레이션 실행

### 실행 단위

TC 단위로 순차 실행한다. 각 TC에 대해:

1. **싱글턴 TC**: Turn 1의 사용자 메시지를 API에 전송, 응답 기록
2. **멀티턴 TC**:
   - 새 세션/대화 시작 (세션 관리 방식에 따라 conversation_id 생성 또는 수신)
   - Turn 1부터 순서대로 사용자 메시지 전송, 각 턴의 응답 기록
   - 이전 턴의 응답이 완료된 후 다음 턴을 전송 (동기 실행)

### 스트리밍 응답 처리

API Spec에 스트리밍이 명시된 경우:
- SSE(Server-Sent Events), WebSocket 등 방식에 맞춰 청크를 누적하여 전체 응답을 조합한다
- 스트림 종료 시점(예: `[DONE]` 이벤트, 연결 종료)까지 대기 후 완전한 응답을 기록한다
- 응답 시간은 첫 청크 수신부터 스트림 종료까지의 전체 시간으로 기록한다

스트리밍 여부가 미확인인 경우:
- 일반(동기) 응답으로 먼저 시도한다
- 응답 헤더에서 스트리밍이 감지되면(Content-Type: text/event-stream 등) 자동으로 스트리밍 처리로 전환한다

### 기록 내용 (턴별)

| 항목 | 내용 |
|------|------|
| 턴 번호 | Turn 1, Turn 2, ... |
| 요청 시각 | ISO 8601 형식 |
| 사용자 메시지 | TC에 정의된 메시지 (그대로 전송) |
| 실제 응답 | 챗봇이 반환한 전체 텍스트 |
| 응답 시간 | 요청~응답 소요 시간 (ms) |
| 상태 | success / error / timeout |
| 에러 내용 | 에러 발생 시 HTTP 상태코드 및 에러 메시지 |

### 에러 처리

- **타임아웃**: 30초 대기 후 timeout 상태로 기록. 해당 TC의 이후 턴은 실행하지 않음
- **API 에러 (4xx/5xx)**: error 상태로 기록. 해당 TC의 이후 턴은 실행하지 않음
- **네트워크 에러**: 1회 재시도 후에도 실패하면 error로 기록

### 진행 표시

TC 실행 중 진행 상황을 PM에게 표시한다:
- `[3/15] TC-003 실행 중... Turn 2/3`
- 에러 발생 시 즉시 알림 (전체 실행은 계속)

## Step 4. 결과 저장

실행 완료 후 `outputs/simulations/{run-id}/`에 결과를 저장한다.

### run-id 규칙
`sim-{YYYYMMDD}-{HHmmss}` (예: `sim-20260410-143052`)

### 디렉토리 구조

```
outputs/simulations/
└── sim-20260410-143052/
    ├── summary.md          # 실행 요약
    ├── tc-001.md           # TC-001 시뮬레이션 결과
    ├── tc-002.md           # TC-002 시뮬레이션 결과
    └── ...
```

### TC별 결과 파일 포맷 (`tc-{번호}.md`)

```markdown
# TC-{번호} 시뮬레이션 결과

## 참조
- **TC**: [TC-{번호}: {제목}](../../testcases/{파일명}.md#tc-{번호})
- **시나리오**: [S-{번호}: {제목}](../../scenarios/{파일명}.md#s-{번호})
- **실행 ID**: {run-id}
- **실행 시각**: {ISO 8601}
- **대화 유형**: {싱글턴 / 멀티턴}

## 실행 결과

| 항목 | 값 |
|------|-----|
| 상태 | {completed / partial / error} |
| 실행 턴 수 | {N} / {전체 턴 수} |
| 총 소요 시간 | {ms} |

## 대화 기록

### Turn 1
- **사용자**: {전송한 메시지}
- **챗봇 응답**: {실제 응답 전문}
- **응답 시간**: {ms}
- **상태**: {success / error / timeout}

### Turn 2
- **사용자**: {전송한 메시지}
- **챗봇 응답**: {실제 응답 전문}
- **응답 시간**: {ms}
- **상태**: {success / error / timeout}

## 평가 대기

이 결과는 아직 평가되지 않았습니다.

### TC 기대 응답 (참조용)
{TC에서 복사한 턴별 기대 응답}

### TC 평가 기준 (참조용)
{TC에서 복사한 평가 기준 체크리스트}
```

### 실행 요약 파일 포맷 (`summary.md`)

## Step 5. PM 결과 보고

실행 완료 후 PM에게 요약을 보고한다:

- 전체 TC 수 및 상태별 분포
- 에러/타임아웃 발생 TC 목록과 원인
- 특이 사항 (예: 예상치 못한 응답 패턴, 스트리밍 자동 전환 등)
- 결과 파일 위치

에러 TC 목록·원인과 특이 사항은 `summary.md`의 `## 에러 및 특이 사항` 섹션에 동일하게 기록되어, 세션 종료 후에도 결과 디렉토리만으로 전체 실행 기록을 복원할 수 있어야 한다.
