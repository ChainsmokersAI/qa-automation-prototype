# /run-simulation — 테스트 시뮬레이션 실행

## 목적
PM이 승인한 TC를 실제 챗봇 API에 실행하여, 챗봇의 실제 응답을 기록한다.
이 스킬은 **실행과 기록**만 담당한다. 기대 응답 대비 평가(합격/불합격 판정)는 별도 단계에서 수행한다.

---

## Step 0. 사전 확인

- `memory/MEMORY.md`를 읽고 이전 세션 상태를 파악한다.
- `context/` 파일들을 읽고 대상 챗봇을 숙지한다.
- `outputs/testcases/`에 승인된 TC가 있는지 확인한다. 없으면 `/generate-testcases`를 먼저 실행하도록 안내한다.
- `context/`에서 API Spec 정보를 확인한다 (Step 1에서 충분성 검증).

---

## Step 1. API Spec 충분성 검증

아래 기준에 따라 context/의 API Spec 정보를 검증한다.

| 등급 | 항목 | 설명 | 미충족 시 대응 |
|------|------|------|---------------|
| **필수** | Endpoint URL | 챗봇 API의 호출 주소 | 시뮬레이션 불가 — PM에게 보충 요청 |
| **필수** | 요청 포맷 | HTTP method, body 구조, 메시지 필드명 | 시뮬레이션 불가 — PM에게 보충 요청 |
| **필수** | 응답 포맷 | 응답 body에서 챗봇 메시지를 추출하는 방법 | 시뮬레이션 불가 — PM에게 보충 요청 |
| **권장** | 인증 방식 | API Key, Bearer Token 등 | 인증 불요 API로 간주. 인증 에러 발생 시 PM에게 요청 |
| **권장** | 세션 관리 | 멀티턴 대화를 위한 conversation_id 관리 방법 | 멀티턴 TC는 실행 불가로 표시(skipped). 싱글턴 TC만 실행 |
| **권장** | 응답 방식 | 일반(동기) 또는 스트리밍(SSE 등). 스트리밍 시 청크 구분 방식 | 일반 응답으로 시도. 응답 헤더에서 스트리밍 감지되면(Content-Type: text/event-stream 등) 자동 전환 |
| **선택** | 파일 업로드 | 이미지/문서 첨부 방법 (multipart 등) | 파일 첨부 관련 TC는 스킵. 텍스트 전용 TC만 실행 |
| **선택** | Rate Limit | 호출 빈도 제한 | 보수적 간격(요청 간 1초 대기)으로 실행 |
| **선택** | 에러 코드 | API 에러 응답 형식과 의미 | HTTP 상태코드로 성공/실패 판단 |

**판정**: 필수 3항목이 모두 있으면 "시뮬레이션 가능", Step 2로 진행. 하나라도 없으면 PM에게 부족 항목을 명시하여 보충을 요청한다 (API 문서, Swagger 스펙, cURL 예시 등 제공 가능). 보충이 완료되면 context/ API Spec 정보를 갱신한 뒤, Step 3의 호출 스크립트 작성을 거쳐 실행에 진입한다.

---

## Step 2. TC 선택

PM에게 실행할 TC 범위를 확인한다:

- **전체 실행**: 모든 승인된 TC 실행
- **시나리오별 실행**: 특정 시나리오의 TC만 실행 (예: S-001의 모든 TC)
- **개별 실행**: 특정 TC만 지정 (예: TC-001, TC-005)

PM이 별도 지정하지 않으면 전체 실행한다.

### 실행 전 요약

선택된 TC 목록을 PM에게 제시한다:
- TC 수, 대화 유형별 분포 (싱글턴 N개, 멀티턴 N개)
- 세션 관리 미확인으로 스킵되는 멀티턴 TC가 있으면 명시
- 예상 API 호출 수 (전체 턴 수 합산)
- **동시 실행 수** (기본 3, Rate Limit 명시 시 자동 하향 — Step 3 병렬 실행 규칙 참조)

PM 확인 후 실행을 시작한다.

---

## Step 3. 시뮬레이션 실행

### 호출 스크립트 작성

실행 전, API Spec을 기반으로 재사용 가능한 호출 스크립트를 작성하고 `outputs/scripts/` 하위에 저장한다. 이후 재실행·디버깅에 활용할 수 있어야 한다.

- **저장 위치**: `outputs/scripts/{chatbot-slug}-api-client.{ext}` — 챗봇 단위 단일 클라이언트. run-id별 복사본을 만들지 않는다.
- **언어**: 에이전트가 Spec에 맞게 판단 (Python 권장 — `requests`/`httpx`/SSE 클라이언트 활용).
- **포함 요소**:
  - 싱글턴·멀티턴 호출 함수 (conversation_id 관리 포함)
  - 스트리밍 응답 자동 감지·누적 처리 (아래 "스트리밍 응답 처리" 규칙과 일치)
  - 인증 헤더 주입
  - **동시 실행 제어**: TC 단위 병렬 실행을 위한 워커 풀/세마포어 (Python 기준 `asyncio.Semaphore` 또는 `concurrent.futures.ThreadPoolExecutor`). 기본 동시 실행 수 3, 파라미터화 가능.
  - **지수 백오프 재시도**: 429 / 503 / `Retry-After` 헤더 / 네트워크 오류 시 지수 백오프로 재시도 (아래 "에러 처리" 규칙과 일치).
  - 에러·타임아웃 처리 (아래 "에러 처리" 규칙과 일치)
- **시크릿**: API Key 등은 스크립트 파일에 하드코딩하지 않고 환경변수나 `inputs/`의 PM 제공 파일에서 읽어온다. 스크립트 상단 주석에 필요한 환경변수 목록을 적는다.
- **이미 존재하는 경우**: 같은 챗봇의 스크립트가 이미 있으면 신규 생성 대신 API Spec 변경 여부를 검증하고, 변경이 있으면 갱신한다.
- **실행 방식**: 에이전트가 Bash로 해당 스크립트를 호출하여 TC를 실행한다.
- **이력 기록**: 신규 생성·갱신 시 `outputs/index.md`의 스크립트 섹션에 한 줄 기록한다.

### 실행 단위

TC 단위로 실행한다 (TC 간에는 병렬, TC 내부 턴은 순차). 각 TC에 대해:

1. **싱글턴 TC**: Turn 1의 사용자 메시지를 API에 전송, 응답 기록
2. **멀티턴 TC**:
   - 새 세션/대화 시작 (세션 관리 방식에 따라 conversation_id 생성 또는 수신)
   - Turn 1 사용자 메시지는 TC 원본 그대로 전송한다.
   - Turn 2 이후: 직전 챗봇 응답이 TC 기대 응답과 **유의미하게 다르면** (챗봇이 되묻거나, 전혀 다른 주제로 답하거나, 다음 턴 메시지의 전제가 깨진 경우) TC 원본 메시지를 그대로 밀어붙이지 말고 **실제 응답 맥락에 맞춰 사용자 메시지를 즉석 조정(re-anchor)한다.**
   - 조정 원칙:
     - TC의 의도(시나리오 목적과 해당 턴에서 확인하려던 핵심 질문)는 유지한다 — 에둘러서라도 TC 목적은 달성한다.
     - 억지로 원본 flow로 끌고 오지 않는다. 챗봇이 되물으면 답해주고, 주제를 바꿨으면 일단 따라간 뒤 자연스럽게 원 질문으로 복귀한다.
     - 조정한 경우 해당 턴의 "비고"에 **조정 사유와 원본 대비 차이**를 반드시 기록한다.
   - 직전 응답이 TC 기대와 충분히 일치하면 TC 원본 메시지를 그대로 전송한다.
   - 이전 턴의 응답이 완료된 후 다음 턴을 전송한다 (동기 실행).

### 병렬 실행 규칙

전체 TC 실행 시간을 줄이기 위해 TC 단위로 병렬 호출한다.

- **병렬 단위**: 서로 다른 TC끼리만 병렬. **같은 TC 내부의 Turn은 반드시 순차** (이전 응답이 다음 턴 메시지의 전제가 되므로).
- **기본 동시 실행 수**: 3. 너무 높으면 Rate Limit에 걸리고, 너무 낮으면 시간 단축 효과가 작다.
- **Rate Limit 반영**:
  - API Spec에 Rate Limit이 명시되어 있으면(예: "60 req/min", "10 rpm per key") **그 한도의 50% 수준**으로 동시 실행 수를 자동 하향한다 (보수적 기본값).
  - Rate Limit이 명시되지 않으면 기본 3을 유지한다.
  - PM이 실행 전 요약에서 동시 실행 수를 명시적으로 조정 요청하면 그에 따른다.
- **세션 관리 주의**: 멀티턴 TC의 conversation_id는 TC별로 독립적으로 관리되어야 한다 (한 세션을 여러 TC가 공유하면 대화가 오염된다).
- **진행 표시**: 병렬 실행 중에는 동시 진행 중인 TC를 `[진행 3/15, 완료 7/15] TC-003·TC-009·TC-011 실행 중` 형태로 집계 표시한다.
- **실행 순서 독립성**: TC는 어느 순서로 완료되어도 결과 파일에 TC 번호대로 기록된다 (저장 시 정렬).

### 스트리밍 응답 처리

API Spec에 스트리밍이 명시된 경우:
- SSE(Server-Sent Events), WebSocket 등 방식에 맞춰 청크를 누적하여 전체 응답을 조합한다
- 스트림 종료 시점(예: `[DONE]` 이벤트, 연결 종료)까지 대기 후 완전한 응답을 기록한다
- 응답 시간은 첫 청크 수신부터 스트림 종료까지의 전체 시간으로 기록한다

스트리밍 여부가 미확인인 경우:
- 일반(동기) 응답으로 먼저 시도한다
- 응답 헤더에서 스트리밍이 감지되면(Content-Type: text/event-stream 등) 자동으로 스트리밍 처리로 전환한다

| 항목 | 내용 |
|------|------|

| 요청 시각 | ISO 8601 형식 |

### 에러 처리 및 재시도

호출 실패 시 원인별로 아래 규칙을 따른다. 재시도는 **해당 턴 범위에서만** 수행하며, 재시도 모두 실패하면 해당 턴을 최종 실패로 기록하고 TC의 이후 턴은 실행하지 않는다.

- **Rate Limit / 일시 서버 오류 (재시도 대상)**:
  - 대상 코드: HTTP 429, 503, 네트워크 타임아웃, 연결 끊김
  - 재시도 횟수: 최대 3회
  - 백오프: 지수 백오프 — 2초 → 4초 → 8초 (jitter ±20% 권장)
  - `Retry-After` 헤더가 응답에 포함되어 있으면 지수 백오프 대신 **헤더 값을 그대로 준수**한다 (초 단위 또는 HTTP-date 둘 다 처리).
  - 429가 반복되면 **전체 실행의 동시 실행 수를 동적으로 절반으로 낮춘다** (예: 3 → 1). 낮춘 값은 이후 TC에도 유지한다.
- **요청 오류 (재시도 안 함)**:
  - 대상 코드: 400, 401, 403, 404, 기타 요청 자체가 잘못된 4xx
  - 재시도해도 같은 결과이므로 즉시 `error` 기록
- **5xx 일반 서버 오류 (500/502/504)**:
  - 1회만 재시도. 실패 시 `error` 기록
- **타임아웃**: 30초 대기 후 timeout. 위 재시도 규칙의 "네트워크 타임아웃"으로 간주하여 1회 재시도 후 `timeout` 기록
- **재시도 기록**: 재시도가 발생한 턴은 "비고"에 `재시도 N회 (429 → backoff {초}s)` 형태로 기록한다.
- **최종 실패 처리**: 재시도 소진 후에도 실패하면 해당 TC는 `partial`(일부 턴만 완료) 또는 `error`(Turn 1부터 실패)로 기록. 해당 TC의 이후 턴은 실행하지 않음.

### 진행 표시

TC 실행 중 진행 상황을 PM에게 표시한다:
- `[3/15] TC-003 실행 중... Turn 2/3`
- 에러 발생 시 즉시 알림 (전체 실행은 계속)

## Step 4. 결과 저장

실행 완료 후 `outputs/simulations/{run-id}/`에 결과를 저장한다.

### run-id 규칙
`sim-{YYYYMMDD}-{HHmmss}` (예: `sim-20260410-143052`)

### 디렉토리 구조

```
outputs/
├── scripts/
│   └── {chatbot-slug}-api-client.py   # 호출 스크립트 (챗봇별 1개, Step 3에서 생성·갱신)
└── simulations/
    └── sim-20260410-143052/
        ├── summary.md      # 실행 요약
        ├── tc-001.md       # TC-001 시뮬레이션 결과
        ├── tc-002.md       # TC-002 시뮬레이션 결과
        └── ...
```

## Step 5. PM 결과 보고

실행 완료 후 PM에게 요약을 보고한다:

- 전체 TC 수 및 상태별 분포
- 에러/타임아웃 발생 TC 목록과 원인
- 특이 사항 (예: 예상치 못한 응답 패턴, 스트리밍 자동 전환 등)
- 결과 파일 위치

에러 TC 목록·원인과 특이 사항은 `summary.md`의 `## 에러 및 특이 사항` 섹션에 동일하게 기록되어, 세션 종료 후에도 결과 디렉토리만으로 전체 실행 기록을 복원할 수 있어야 한다.
