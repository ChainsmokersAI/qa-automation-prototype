# /generate-testcases — 테스트 케이스 생성

## 목적
PM이 승인한 시나리오를 기반으로, 실행 가능한 수준의 구체적인 테스트 케이스를 생성한다.
각 TC는 명확한 대화 흐름(싱글턴 또는 멀티턴), 턴별 기대 응답, 전체 평가 기준을 포함하여 챗봇 품질을 객관적으로 검증할 수 있어야 한다.

---

## Step 0. 사전 확인

- `memory/MEMORY.md`를 읽고 이전 세션 상태를 파악한다.
- `context/` 파일들을 읽고 대상 챗봇을 숙지한다.
- `outputs/scenarios/`에 승인된 시나리오가 있는지 확인한다. 없으면 `/generate-scenarios`를 먼저 실행하도록 안내한다.

---

## Step 1. PM 명시 질의 (3종) — 필수 선행 조건

> **⚠️ 필수 선행 조건 (생략 불가)**
>
> 이 Step은 PM이 아무리 급해 보여도, context가 아무리 풍부해도, 이전 세션에서 이미 유사 질의를 했어도 **절대 생략하지 않는다.** 아래 3가지 답을 모두 PM으로부터 받기 전에는 Step 2(TC 작성)로 진입할 수 없다.
>
> - [ ] **대상 시나리오** 확정
> - [ ] **시나리오별 TC 개수** 확정
> - [ ] **TC 대화 Turn 수** 확정
>
> **진입 체크포인트**: Step 2 첫 줄에 "Step 1에서 확정된 3가지 값을 명시적으로 복창한 뒤 작성을 시작한다"는 규칙이 박혀 있다. 질의를 건너뛰면 복창할 값이 없어 작성 자체가 시작될 수 없다.
>
> **예외 없음**: "시간이 부족해 보여서", "PM이 이미 대충 말한 것 같아서", "이전에 한 번 물었으니까" 같은 이유로 생략 금지. PM이 이전에 말한 값이 있으면 **그 값을 옵션에 미리 채워 제시**하되, 질의 자체는 여전히 수행한다(= 복창 확인).

PM에게 아래 3가지를 **한 번의 `AskUserQuestion` 호출**로 묶어 제시한다 (tool은 최대 4 questions 지원). 호출이 부적절한 상황이면 평문 메시지로 동일 3개를 한 번에 나열한다.

### 1) 대상 시나리오

| 옵션 | 의미 |
|------|------|
| 전체 | 모든 승인된 시나리오에 대해 TC 생성 |
| 중요도 상만 | 중요도 상 시나리오만 |
| 중요도 상+중 | 상+중 시나리오 |
| 특정 ID 지정 | PM이 S-001, S-003 등 명시 (AskUserQuestion의 "Other" 경로) |

**기본값(PM이 "맡김" 선택 시)**: 전체 — 시나리오에 부여된 중요도 상 → 중 → 하 순으로 생성.

### 2) 시나리오별 TC 개수

| 옵션 | 의미 |
|------|------|
| 소 (1~2개) | 핵심만 빠르게 |
| 중 (3~5개) | 균형 잡힌 커버리지 (기본값) |
| 대 (6~8개) | 엣지 케이스까지 |
| 시나리오별 자동 | 시나리오 복잡도에 따라 에이전트가 개별 판단 |

**기본값**: 중 (3~5개).

### 3) TC 대화 Turn 수

| 옵션 | 의미 |
|------|------|
| 싱글턴 | 1턴만 |
| 짧게 (3~5턴) | 짧은 멀티턴 |
| 기본 (5~10턴) | 기본 범위 (기본값) |
| 길게 (10턴+) | 다단계 작업·복잡한 흐름 |
| 시나리오별 자동 | 시나리오 성격에 따라 에이전트가 개별 판단 |

**기본값**: 기본 (5~10턴). User Logs가 있으면 실제 평균 턴 수를 병기하여 PM에게 제시하고, PM이 "User Logs 기반"을 선택하면 실제 값 우선.

### 질의 후 상태 기록

PM 답변을 받으면 아래 3줄을 에이전트 작업 맥락에 명시적으로 고정한다 (Step 2 진입 체크포인트에서 그대로 복창한다):

```
- 대상 시나리오: {…}
- 시나리오별 TC 개수: {…}
- TC 대화 Turn 수: {…}
```

## Step 2. TC 작성

> **🔒 진입 체크포인트**: 작성 시작 전, Step 1에서 확정된 3가지 값을 아래 형식으로 **반드시 복창**한다. 복창할 값이 없으면 Step 1으로 돌아간다. 예외 없음.
>
> ```
> - 대상 시나리오: {…}
> - 시나리오별 TC 개수: {…}
> - TC 대화 Turn 수: {…}
> ```
>
> 이 복창이 없는 상태로 TC 작성을 시작하는 것은 이 스킬의 규약 위반이다.

각 시나리오에 대해 아래 포맷으로 TC를 작성한다. 하나의 시나리오에서 여러 TC가 파생된다 (개수는 Step 1에서 확정한 값을 따른다).

```markdown
## TC-{번호}: {TC 제목}

### 시나리오
[S-{번호}: {시나리오 제목}](../scenarios/{파일명}.md#s-{번호})

### 관련 컨텍스트
- [{항목명}](../../context/{파일명}.md#{섹션-앵커})
- [{항목명}](../../context/{파일명}.md#{섹션-앵커})

### 대화 유형
{싱글턴 / 멀티턴}

### 대화 흐름

#### Turn 1
- **사용자**: {챗봇에게 보내는 구체적인 메시지. 실제 사용자가 보낼 법한 자연어}
- **기대 응답**:
  - {챗봇이 응답해야 할 핵심 내용 1}
  - {챗봇이 응답해야 할 핵심 내용 2}

#### Turn 2
- **사용자**: {이전 턴의 응답을 받은 후 이어지는 사용자 메시지}
- **기대 응답**:
  - {핵심 내용}

<!-- 필요한 만큼 Turn N 추가 -->

### 평가 기준
- [ ] {전체 대화에 대한 구체적이고 객관적인 기준 1}
- [ ] {구체적이고 객관적으로 판단 가능한 기준 2}
```

### 작성 규칙

- **관련 컨텍스트**: context/ 파일의 **섹션 레벨**까지 링크한다. 특히 평가 기준의 근거가 되는 구체적 수치, 정책, 스펙을 참조한다.
- **대화 유형 결정**:
  - **Step 1에서 확정된 "TC 대화 Turn 수"를 따른다.** 싱글턴이 지정되면 모든 TC는 싱글턴, 특정 범위가 지정되면 그 범위 내에서 작성.
  - Step 1에서 "시나리오별 자동"을 선택한 경우에만 에이전트가 시나리오 성격에 따라 개별 판단한다 (기본 멀티턴 5~10턴, 챗봇 유형별 참고: 상담·추천·튜터 5~8턴, 에이전트형 10턴+, FAQ도 기본 멀티턴).
  - **User Logs가 있으면 실제 대화 패턴을 최우선으로 반영한다** — 실제 로그의 평균 턴 수, 턴 전환 패턴, 사용자 말투를 TC에 그대로 녹인다 (Step 1에서 User Logs 기반을 선택한 경우 필수).
  - 싱글턴 TC는 Turn 1만 있는 동일 포맷이다 (별도 형식 없음).
- **사용자 메시지**: 실제 사용자가 쓸 법한 자연어로 작성한다. 구어체·비문·오타 등 현실적 표현 포함 가능.
- **기대 응답**: 정답 문장이 아니라 **포함해야 할 핵심 내용**을 나열한다. 챗봇이 동일한 의미를 다른 표현으로 전달해도 통과할 수 있도록.
- **이어지는 사용자 메시지**: 이전 턴의 기대 응답을 전제로 자연스럽게 이어지는 메시지를 작성한다. 실제 대화처럼 이전 맥락을 참조하는 표현을 사용한다 (예: "그거 말고 다른 건?", "방금 말한 상품 가격이 얼마야?").
- **턴 수**: Step 1에서 PM이 지정한 턴 수 범위를 따른다. 범위를 크게 벗어나는 TC는 작성 이유를 짧게 명시한다.
- **시나리오별 TC 개수**: Step 1에서 확정된 개수(소/중/대/자동)를 시나리오마다 일관되게 적용한다. "자동"인 경우에만 시나리오 복잡도에 따라 개별 조정.
- **평가 기준**: 예/아니오로 판단 가능한 체크리스트 형태로 작성한다. "적절한 응답을 했는가" 같은 모호한 기준은 사용하지 않는다. 멀티턴 TC의 경우, 전체 대화 흐름의 일관성도 평가 항목에 포함한다.
- **TC ID**: TC-001부터 순차 부여. 기존 TC가 있으면 이어서 번호 부여.

---

## Step 3. PM 확인

생성된 TC를 시나리오별로 그룹핑하여 PM에게 제시한다.

- 각 TC의 제목 + 대화 유형 + 대화 흐름 요약(턴 수, 첫 사용자 메시지) + 평가 기준 요약을 리스트로 먼저 제시
- PM 피드백 반영: TC 수정, 삭제, 추가, 평가 기준 조정

---

## Step 4. TC 저장

PM 확인이 완료되면 `outputs/testcases/`에 시나리오별로 저장한다.

```
outputs/testcases/
├── s-001-testcases.md    # S-001의 모든 TC
├── s-002-testcases.md    # S-002의 모든 TC
└── ...
```

- `outputs/index.md`에 생성 이력 기록

---

## Step 5. 메모리 업데이트

- `memory/task-log.md`에 작업 기록 (생성된 TC 수, 시나리오 커버리지)
- `memory/lessons-learned.md`에 교훈 기록 (있을 경우)
- `memory/MEMORY.md`의 프로젝트 현황 갱신
